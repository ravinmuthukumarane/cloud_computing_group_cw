Correct successful command path
===============================

Step 1 - Create k3d cluster without Traefik
-------------------------------------------

The first cluster had Traefik enabled by default, which conflicted with NGINX
Ingress. The working cluster was created with Traefik disabled.

Command:

k3d cluster create cloud-salary-cluster \
  --servers 1 \
  --agents 0 \
  --port "8080:80@loadbalancer" \
  --port "8443:443@loadbalancer" \
  --k3s-arg "--disable=traefik@server:0"

What it does:

Creates a single-node k3d cluster.
Maps VM port 8080 to the Kubernetes load balancer HTTP port 80.
Maps VM port 8443 to the Kubernetes load balancer HTTPS port 443.
Disables Traefik so NGINX Ingress owns ingress traffic.

Verify:

sudo kubectl get nodes

Expected:

k3d-cloud-salary-cluster-server-0 should be Ready.

Mistakes and corrections
========================

Mistake 1 - Traefik conflicted with NGINX Ingress
------------------------------------------------

What happened:

The first k3d cluster was created without disabling Traefik.
k3s installs Traefik by default.

Symptoms:

curl http://localhost:8080/

returned:

404 page not found

Checking controllers showed:

traefik pod Running
svclb-traefik Running
ingress-nginx-controller Running
svclb-ingress-nginx-controller Pending

Diagnosis:

Port 8080 traffic was reaching Traefik, not NGINX Ingress.

Correction:

The cluster was recreated with Traefik disabled:

k3d cluster create cloud-salary-cluster \
  --servers 1 \
  --agents 0 \
  --port "8080:80@loadbalancer" \
  --port "8443:443@loadbalancer" \
  --k3s-arg "--disable=traefik@server:0"

Result:

NGINX Ingress handled traffic correctly.
curl http://localhost:8080/ returned frontend HTML.


Mistake 2 - App pods tried to read a Secret from the data namespace
-------------------------------------------------------------------

What happened:

The identity Deployment originally referenced postgres-secret for DB_PASSWORD.
postgres-secret existed only in the data namespace.
The identity pod runs in the app namespace.

Symptom:

identity pod status:

CreateContainerConfigError

Diagnosis:

Kubernetes Secrets are namespace-scoped.
Pods in app cannot read Secrets from data.

Correction:

Created:

k8s/secrets/app-db-secret.yaml

with:

namespace: app

and key:

DB_PASSWORD

Updated application deployments to use:

app-db-secret

Affected deployments:

identity
salary-submission
vote
search
stats

Result:

identity pod became 1/1 Running.
Other backend services also ran successfully.


Mistake 3 - Frontend was built with Docker Compose API URL
---------------------------------------------------------

What happened:

The frontend image was first built with:

VITE_API_BASE_URL=http://localhost:5050/api

This is suitable for local Docker Compose when the browser talks to the BFF on
host port 5050.

In Kubernetes ingress, frontend should call:

/api

because the same ingress routes /api to BFF.

Symptom:

Checking the built JS showed:

http://localhost:5050/api

Command used:

curl -s http://localhost:8080/assets/index-DWr9rItW.js | grep -o "http://localhost:5050/api\|/api" | head

Correction:

Rebuilt frontend with:

sudo docker build --no-cache \
  --build-arg VITE_API_BASE_URL=/api \
  -t frontend-service:k8s-1.0.0 \
  frontend

Then pushed as:

nipuniamarasinghe/frontend-service:1.0.1

Updated:

k8s/frontend/frontend-deployment.yaml

Result:

Frontend pod rolled out successfully using image:

nipuniamarasinghe/frontend-service:1.0.1


Mistake 4 - Docker cache reused the old frontend build
------------------------------------------------------

What happened:

The frontend was rebuilt with a different build arg, but Docker reused the cached
Vite build layer:

CACHED [build 8/8] RUN npm run build

Symptom:

The JS bundle still contained:

http://localhost:5050/api

Correction:

Used:

--no-cache

and updated the Dockerfile build step to print the build arg:

RUN echo "Building frontend with VITE_API_BASE_URL=$VITE_API_BASE_URL" && npm run build

Result:

Build output showed:

Building frontend with VITE_API_BASE_URL=/api


Mistake 5 - Reusing the same image tag made Kubernetes pull/cache confusing
---------------------------------------------------------------------------

What happened:

The frontend Deployment was restarted while still using an old or unavailable
image tag.

Symptoms:

Rollout stuck at:

Waiting for deployment "frontend" rollout to finish: 1 old replicas are pending termination...

Frontend pod showed:

ErrImagePull

Deployment showed image:

nipuniamarasinghe/frontend-service:k8s-1.0.0

Correction:

Used a new image tag:

nipuniamarasinghe/frontend-service:1.0.1

Updated:

k8s/frontend/frontend-deployment.yaml

with:

imagePullPolicy: Always

Result:

Frontend rollout completed successfully.


Mistake 6 - Tried to map port 80 to 8080 using iptables
-------------------------------------------------------

What happened:

An attempt was made to access the app without :8080 by redirecting host port 80
to 8080 using iptables.

Rules added:

sudo iptables -t nat -A PREROUTING -p tcp --dport 80 -j REDIRECT --to-port 8080
sudo iptables -t nat -A OUTPUT -p tcp -o lo --dport 80 -j REDIRECT --to-port 8080

Symptom:

Traffic became unreliable and neither access path worked cleanly.

Diagnosis:

Docker already had NAT rules mapping:

8080 -> k3d load balancer port 80

Adding extra redirect rules caused messy host-level NAT behavior.

Correction:

Decision was made to forget direct port 80 mapping and continue with the working
port:

http://VM_PUBLIC_IP:8080/

Recommended future clean approach if direct port 80 is required:

Recreate k3d with:

--port "80:80@loadbalancer"

instead of using iptables.
