SECTION 1 - COURSEWORK REQUIREMENTS
=========================================================================

Module:
- 7BUIS027C.2 – Cloud Computing Applications 2026

Submission:
- Report and viva
- Due date: 16 April 2026, 23:59 IST

Core system to build:
- A cloud-native microservice application for salary transparency in Sri Lanka
- Similar to TechPays / salary transparency platforms

Required workflow:
- submission -> voting -> approval -> search -> stats

Non-negotiable functional rules:
- Users can anonymously submit salary information
- Users can search salaries by country, company, role, and experience level
- Users can vote on whether submitted salaries are trustworthy
- Login is required only for community actions such as upvoting/downvoting/reporting
- Salary submissions must remain anonymous
- Identity data must be stored separately from salary data
- Salary records must not be directly linked to user emails

Required architecture:
- Frontend service
- BFF service (Backend-for-Frontend)
- Salary-submission service
- Identity service
- Vote service
- Search service
- Stats service
- PostgreSQL service

Cloud-native requirements:
- All services containerized using Docker
- Deployed to a single-node Kubernetes cluster on Azure
- External traffic routed through load balancer and Kubernetes Ingress
- Application microservices run in the app namespace
- PostgreSQL runs in the data namespace
- Services should be stateless
- Configuration must be externalized using ConfigMaps and Secrets
- Readiness/liveness behavior should be shown
- PostgreSQL must use persistent storage (PVC)

Database requirements:
- One PostgreSQL instance
- Separate schemas:
  - identity
  - salary
  - community

Schema expectations:
- identity schema stores user and authentication data
- salary schema stores raw submissions and approved data
- community schema stores votes and optionally reports

Security and privacy requirements:
- Passwords must be hashed, never stored in plain text
- Voting/reporting endpoints must require authentication
- Internal services must not be exposed publicly
- Only frontend/BFF should be exposed via ingress
- An anonymize toggle must be implemented
- Public records must hide/generalize identifying information when anonymize is enabled

Evidence required:
- Screenshots and/or terminal outputs proving:
  - salary submission stored as PENDING
  - user signup/login and token/session creation
  - vote recording
  - submission becoming APPROVED when threshold is reached
  - approved salary appearing in search
  - stats updating based on approved salaries

Repository/readme expectations:
- Complete reproducible Kubernetes manifests:
  - namespaces
  - deployments
  - services
  - ingress
  - persistent volume resources for PostgreSQL
- README must contain exact commands to:
  - build images
  - apply manifests
  - initialize DB schema
  - test workflow
- No hardcoded secrets or config in source code

SECTION 2 - AGREED IMPLEMENTATION DECISIONS
=========================================================================

Do NOT use Kafka.
- Kafka is not required by the coursework
- It adds unnecessary complexity
- Simple synchronous HTTP communication is enough

Recommended stack:
- Frontend: React
- Backend services: Node.js + Express
- Database: PostgreSQL
- Auth: JWT
- Password hashing: bcrypt
- Containers: Docker
- Orchestration: Kubernetes
- Target cloud: Azure

Important development order:
1. Design first
2. Build working code locally
3. Dockerize early after core flow works
4. Run in local Kubernetes
5. Deploy to Azure

Do not:
- start directly in Azure
- start with Kubernetes before application logic works
- leave Docker/Kubernetes to the very end
- overengineer the system

SECTION 3 - SYSTEM DESIGN
=========================================================================

Architecture flow:
Frontend -> BFF -> Identity Service
                -> Salary Submission Service
                -> Vote Service
                -> Search Service
                -> Stats Service

Service responsibilities:

1. Frontend Service
- Provides UI
- Sends API requests only to BFF
- Stores token after login
- Displays search and stats

2. BFF Service
- Single public API entry point
- Routes requests to internal services
- Validates JWT on protected endpoints
- Prevents direct exposure of internal services

3. Identity Service
- Signup
- Login
- Password hashing
- JWT generation
- Optional token validation

4. Salary Submission Service
- Anonymous salary submission
- Input validation
- Store submission as PENDING
- Store anonymize flag
- No user identity data here

5. Vote Service
- Authenticated voting only
- One vote per user per submission
- Track votes
- Apply approval rule
- Mark submission APPROVED when threshold reached

6. Search Service
- Return only APPROVED salaries
- Filter by country, company, role, experience level
- Apply anonymization rules

7. Stats Service
- Return aggregated salary stats
- Average
- Median
- Percentiles
- Use only APPROVED salaries

SECTION 6 - DATABASE DESIGN
=========================================================================

Schemas:
- identity
- salary
- community

Tables:

identity.users
- id UUID PRIMARY KEY
- email UNIQUE NOT NULL
- password_hash NOT NULL
- created_at
- updated_at

salary.submissions
- id UUID PRIMARY KEY
- country NOT NULL
- company NOT NULL
- role NOT NULL
- experience_level NOT NULL
- salary_amount NOT NULL
- currency NOT NULL
- anonymize BOOLEAN DEFAULT TRUE
- status VARCHAR DEFAULT PENDING
- created_at
- updated_at

community.votes
- id UUID PRIMARY KEY
- submission_id NOT NULL
- user_id NOT NULL
- vote_type NOT NULL
- created_at
- UNIQUE(submission_id, user_id)

community.reports (optional)
- id UUID PRIMARY KEY
- submission_id NOT NULL
- user_id NOT NULL
- reason NOT NULL
- created_at

Important privacy rule:
- no email in salary tables
- salary records must not expose identity linkage publicly

Approval rule:
- Keep it simple and explicit
- Example:
  - upvotes >= 5 => submission becomes APPROVED

Public search rule:
- Only APPROVED salaries are visible

Stats rule:
- Only APPROVED salaries are used for aggregates

SECTION 12 - STARTER PROMPT FOR COPILOT
=========================================================================

Use the prompt below to start the project.

PROMPT START
------------
I am building a coursework project for a Cloud Computing module. I need a cloud-native microservice application for salary transparency in Sri Lanka.

Use this stack:
- React for frontend
- Node.js + Express for backend services
- PostgreSQL for database
- JWT for authentication
- bcrypt for password hashing

Architecture:
- frontend
- bff
- identity service
- salary-submission service
- vote service
- search service
- stats service
- one PostgreSQL instance with schemas: identity, salary, community

Important rules:
- frontend must call only the BFF
- salary submission is anonymous and does not require login
- login is required only for voting/reporting
- salary data must be separated from identity data
- salary records must not be linked publicly to user emails
- all public traffic should go through the BFF
- all services should be designed to be stateless

Implement the project incrementally.

First, generate the backend foundation with this folder structure:
- services/identity
- services/salary-submission
- services/vote
- services/search
- services/stats
- services/bff
- database/init
- frontend

Start by creating:
1. PostgreSQL SQL initialization files:
   - 001_create_schemas.sql
   - 002_create_tables.sql
   - 003_create_indexes.sql

2. Identity service:
   - POST /signup
   - POST /login
   - hash passwords with bcrypt
   - return JWT on login
   - use environment variables for config

3. Salary submission service:
   - POST /submissions
   - accept anonymous salary submission
   - store records as PENDING
   - include anonymize flag
   - validate required fields

4. Vote service:
   - POST /votes
   - require authenticated userId from BFF
   - only one vote per user per submission
   - if upvotes >= 5, mark salary.submissions.status as APPROVED

5. Search service:
   - GET /search
   - return only APPROVED salary records
   - support filters by country, company, role, experience level
   - if anonymize=true, return company as Anonymous

6. Stats service:
   - GET /stats
   - return count, average, median, p90
   - use only APPROVED salaries

7. BFF service:
   - expose public routes:
     - POST /api/auth/signup
     - POST /api/auth/login
     - POST /api/salaries
     - POST /api/votes
     - GET /api/search
     - GET /api/stats
   - proxy requests to internal services
   - verify JWT for protected endpoints such as voting

Requirements:
- use clean modular Express structure
- create package.json for each service
- add .env.example for each service
- add basic error handling
- add /health endpoint for each service
- keep code simple and coursework-focused
- do not use Kafka
- do not overengineer
- prepare the structure so Dockerization and Kubernetes can be added next

Generate the code step by step, starting with the SQL files and the identity service.
PROMPT END
