SECTION 1 - COURSEWORK BIG PICTURE
======================================================================

Project:
Cloud Salary Transparency System

Goal:
Build and deploy a cloud-native microservice application for salary transparency.

Required workflow:
submission -> voting -> approval -> search -> stats

Core functional rules:
- users can anonymously submit salary information
- users can search salaries by country, company, role, and experience level
- users can vote on salary submissions only when logged in
- login is required only for community actions such as upvote/downvote/report
- salary submissions must remain anonymous
- identity data must be stored separately from salary submission data
- salary records must not be directly linked to user emails

Required services:
- frontend service
- BFF service
- identity service
- salary-submission service
- vote service
- search service
- stats service
- PostgreSQL service

Required architecture principles:
- frontend talks only to the BFF
- internal services are not exposed publicly
- all services are containerized
- services run in Kubernetes
- application services run in namespace: app
- PostgreSQL runs in namespace: data
- external traffic is routed through ingress / load balancer
- configuration is externalized using ConfigMaps and Secrets
- services are stateless
- PostgreSQL uses persistent storage
- readiness/liveness behavior should be shown

Current state of the project:
- frontend has already been developed
- backend services have already been developed separately
- services have been containerized
- Docker images have been pushed to Docker Hub
- next phase is Kubernetes deployment
- later phase is Azure Pipeline CI/CD automation

Deployment environment decision:
- use a single-node Kubernetes cluster
- use an Azure VM as the machine that runs the cluster
- use k3d on the Azure VM
- use NGINX Ingress
- deploy frontend as its own service
- React frontend will be served using NGINX inside the frontend container
- public routing will be:
  - /      -> frontend service
  - /api/* -> BFF service

Architecture request flow:
Browser -> Ingress
  /      -> Frontend service
  /api/* -> BFF service -> identity / salary-submission / vote / search / stats

This means:
- browser loads the frontend from the frontend service
- frontend JavaScript makes API requests to /api/*
- ingress routes /api/* to the BFF
- BFF talks to the internal backend services

SECTION 4 - TARGET KUBERNETES ARCHITECTURE
======================================================================

Namespaces:
- app
- data

app namespace contains:
- frontend
- bff
- identity
- salary-submission
- vote
- search
- stats

data namespace contains:
- postgres
- postgres persistent volume claim

Traffic routing:
- ingress controller: NGINX Ingress
- / routes to frontend service
- /api routes to bff service

Frontend deployment model:
- React build files are served by NGINX inside the frontend container
- frontend remains its own Kubernetes Deployment and Service
- do not manually copy build files into a default cluster nginx
- frontend must remain a proper frontend service in the architecture

Backend deployment model:
- each backend service gets:
  - its own Deployment
  - its own ClusterIP Service

PostgreSQL deployment model:
- postgres runs in data namespace
- postgres gets:
  - Deployment or Stateful-style setup
  - Service
  - PersistentVolumeClaim
- database initialization should use the SQL files already created
- DB credentials should come from Kubernetes Secret
- DB config should not be hardcoded in source code

Config approach:
- ConfigMap for non-sensitive values
- Secret for sensitive values

Examples of non-sensitive values:
- service ports
- database host
- database name
- upstream URLs
- environment mode

Examples of sensitive values:
- postgres password
- JWT secret
- any access tokens or passwords

Health checks:
- use existing /health endpoints for readiness/liveness if already available

--------------------------------------
