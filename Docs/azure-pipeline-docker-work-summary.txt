Important VM note
-----------------

This work was done on a Debian VM.

Docker commands need sudo on this VM because the current Linux user does not
have permission to access the Docker socket:

/var/run/docker.sock

Use:

sudo docker ...

Without sudo, Docker commands fail with a permission error.

Examples:

sudo docker ps
sudo docker images
sudo docker build ...
sudo docker compose up ...


Container vs image reminder
---------------------------

Images are build artifacts.
Containers are running or stopped instances of images.

Useful commands:

sudo docker images
sudo docker ps
sudo docker ps -a

Remove a container:

sudo docker rm CONTAINER_NAME

Remove an image:

sudo docker rmi IMAGE_NAME:TAG

Stop a container:

sudo docker stop CONTAINER_NAME

docker stop does not take image tags.

Correct:

sudo docker stop bff-service

Incorrect:

sudo docker stop bff-service:1.0.0

Important Docker Hub + sudo note
--------------------------------

The Docker Hub login for nipuniamarasinghe was stored in:

/home/ravinmuthukumarane/.docker-nipuniamarasinghe/config.json

Because Docker commands require sudo on this VM, a normal command like this can
fail during push:

sudo docker push nipuniamarasinghe/search-service:1.0.0

It can fail because sudo uses root's Docker config, not the custom user config.

Use this instead:

sudo env DOCKER_CONFIG=/home/ravinmuthukumarane/.docker-nipuniamarasinghe \
  docker push nipuniamarasinghe/search-service:1.0.0

The same pattern applies for all pushes to the nipuniamarasinghe Docker Hub
account.

Dockerfile pattern
------------------

Most services use this pattern:

FROM node:20-alpine

WORKDIR /app

ENV NODE_ENV=production

COPY package*.json ./
RUN npm ci --omit=dev && npm cache clean --force

COPY src ./src

USER node

EXPOSE SERVICE_PORT

CMD ["npm", "start"]

The BFF Dockerfile currently uses the same Node Alpine base and production npm
install, but does not yet include USER node or npm cache clean:

services/bff/Dockerfile

This still works. It can be aligned later if desired.


Why node:20-alpine was used
---------------------------

The Debian VM is the host operating system.

The Docker image still needs its own filesystem and runtime. For Node services,
the image needs Node.js and npm inside the container.

node:20-alpine was used because it is an official Node image and is smaller than
full Debian-based Node images.

FROM scratch was discussed but not used because these services are Node.js apps.
FROM scratch is not suitable unless a fully static binary is being copied in,
which is usually for Go/Rust/C apps, not Node.

PostgreSQL official image
-------------------------

The official PostgreSQL image used:

postgres:16-alpine

The local database configuration used by the services:

DB_HOST=localhost
DB_PORT=5432
DB_NAME=cloud_salary
DB_USER=postgres
DB_PASSWORD=65610

When running services inside Docker Compose, DB_HOST is overridden to:

DB_HOST=postgres

This is required because localhost inside a container means the same container,
not the host machine and not another container.


SQL initialization files
------------------------

These SQL files are required to initialize the database:

Docs/SQL Files/001_create_schemas.sql
Docs/SQL Files/002_create_tables.sql
Docs/SQL Files/003_create_indexes.sql

They are mounted into the official Postgres container at:

/docker-entrypoint-initdb.d

In docker-compose.yml:

"./Docs/SQL Files:/docker-entrypoint-initdb.d:ro"

The Postgres image automatically runs files in /docker-entrypoint-initdb.d only
when the database directory is empty.

This means SQL init scripts run only on first database initialization.

To force the scripts to run again with Docker Compose, remove the Compose volume:

sudo docker compose down -v

Warning:

docker compose down -v deletes the Compose PostgreSQL database volume.


PostgreSQL standalone commands used before Compose
--------------------------------------------------

Pull official Postgres image:

sudo docker pull postgres:16-alpine

Create standalone database volume:

sudo docker volume create cloud_salary_pgdata

Run standalone Postgres:

sudo docker run -d \
  --name cloud-salary-postgres \
  -e POSTGRES_DB=cloud_salary \
  -e POSTGRES_USER=postgres \
  -e POSTGRES_PASSWORD=65610 \
  -p 5432:5432 \
  -v cloud_salary_pgdata:/var/lib/postgresql/data \
  -v "$(pwd)/Docs/SQL Files:/docker-entrypoint-initdb.d:ro" \
  postgres:16-alpine

Check Postgres:

sudo docker exec cloud-salary-postgres pg_isready -U postgres -d cloud_salary

Open psql:

sudo docker exec -it cloud-salary-postgres psql -U postgres -d cloud_salary

Inside psql:

\dn
\dt identity.*
\dt salary.*
\dt community.*
\di identity.*
\di salary.*
\di community.*

Docker Hub repository note
--------------------------

If push fails with:

push access denied, repository does not exist or may require authorization

Then either:

1. The Docker CLI is using the wrong Docker config/login.
2. The repository does not exist under that Docker Hub namespace.
3. The account does not have permission to push to that namespace.

For nipuniamarasinghe, use:

sudo env DOCKER_CONFIG=/home/ravinmuthukumarane/.docker-nipuniamarasinghe \
  docker push nipuniamarasinghe/IMAGE_NAME:1.0.0

If it still fails, create the repository manually in Docker Hub under the
nipuniamarasinghe account.

Cleaning old standalone containers before Compose
-------------------------------------------------

Before running Compose fresh, stop and remove standalone containers:

sudo docker stop vote-service stats-service search-service identity-service bff-service cloud-salary-postgres

sudo docker rm vote-service stats-service search-service identity-service bff-service cloud-salary-postgres

If some containers do not exist, Docker may print "No such container". That is
fine.


Run Docker Compose in foreground
--------------------------------

To see all initialization logs, do not use -d.

Run:

sudo docker compose up --build

This shows:

1. Image build output.
2. Postgres startup.
3. SQL initialization logs from /docker-entrypoint-initdb.d.
4. Node service startup logs.
5. Any connection or configuration errors.

Stop foreground Compose:

Ctrl + C

Then clean up containers:

sudo docker compose down

Reset Compose database and rerun SQL init scripts:

sudo docker compose down -v
sudo docker compose up --build


Run Docker Compose in detached mode
-----------------------------------

Use this when logs do not need to stay attached:

sudo docker compose up -d --build

Check services:

sudo docker compose ps

Show logs:

sudo docker compose logs

Show logs for one service:

sudo docker compose logs bff
sudo docker compose logs postgres

Stop:

sudo docker compose down

Common CI/CD tasks will be:

3. Tag images with a stable version, commit SHA, or build ID.

5. Push images to Docker Hub.

Recommended Docker Hub secrets for Azure Pipelines:

DOCKERHUB_USERNAME
DOCKERHUB_TOKEN

Use a Docker Hub access token instead of a real account password.

Do not store Docker Hub passwords or tokens directly in repo files.

Files that should not be baked into images
------------------------------------------

.env files are excluded by .dockerignore.

Runtime environment values are provided by:

1. docker run --env-file
2. docker-compose.yml env_file and environment
3. Later, Azure Pipeline variables or deployment environment variables

Do not bake secrets into Docker images.
